# 流程：错误恢复

## 错误分类

| 类型 | 示例 | 默认策略 |
|---|---|---|
| 节点找不到 | 等待"搜索"按钮超时 | recovery 配置（fallback_to_llm / prompt_user / abort） |
| 步骤超时 | 网络慢导致页面没出来 | retry 1 次后 abort |
| LLM 网络错误 | 超时 / 5xx | 退避重试 3 次 |
| LLM 返回非法 JSON | 模型不稳定 | 重试 1 次后兜底文字 |
| RiskGuard 触发 | 检测到付款页 | 暂停等待用户确认 |
| App 闪退 / 系统返回 | 应用崩了 | 重启该步骤一次后 abort |
| 用户主动取消 | 点停止 | 立即取消协程 |
| 全场景超时 | 60s 没完成 | 提示用户并展示进度 |

## 恢复策略

### `fallback_to_llm`

把以下信息打包发给 LLM：

- 当前屏幕语义摘要
- 已执行的步骤
- 剩余的步骤
- 错误描述

期待 LLM 返回新的 tool_calls 续航。

### `prompt_user`

弹浮窗：

- "卡在 [步骤描述]，你来接管一下？"
- 选项：继续等 / 我来操作 / 放弃

### `abort`

- 写 habit_events（outcome=failed）
- 浮窗变灰，提示"任务终止"
- 提供"再试一次" / "反馈问题" 按钮

## 用户视角

- 失败时浮窗状态条显示具体哪一步失败
- 不出现裸的 stack trace
- "反馈问题"会打包匿名日志（用户确认后）

## 日志

- 每个失败都写 ERROR 级日志
- 包含：step id / 期望状态 / 实际状态 / 时间戳
- 7 天保留

## 测试

- mock 节点不存在的场景
- mock 网络断
- mock LLM 5xx
- mock 用户取消

## 验收

- 每种错误类型都能优雅降级，不崩溃
- 失败信息对用户友好
- 日志可用于事后定位
